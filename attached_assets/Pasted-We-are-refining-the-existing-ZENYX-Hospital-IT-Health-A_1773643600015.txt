QUESTION DISPLAY LOGIC
Questions should not all be shown to all users.

For example:
- Multi-speciality hospitals should see more complex infra, uptime, branch, and monitoring questions
- Clinics should see simpler operational and continuity questions
- Diagnostic centres should see imaging/reporting/data-flow questions

Role examples:
- IT Head / System Admin → deeper technical questions
- Owner / Director → business continuity, downtime, reporting, confidence questions
- Doctor / Department Head → patient workflow continuity, report access, printer/system issues
- Procurement → lifecycle planning, maintenance, replacement planning
- Operations Manager → delays, billing, patient records availability, department complaints

QUESTION STYLE RULES
Use simple, understandable questions.
Avoid technical jargon in the main question.

Use this structure:
1. Short main question
2. One line helper text below when needed
3. Optional “Why this matters” or tooltip for complex questions

Example:
Main question:
“Do you get alerts when servers, internet, or key systems go down?”
Helper:
“This helps your team respond before downtime affects hospital work.”

Do NOT make every question long and overloaded.
Keep main questions short.

ANSWER UI RULES
Use side-by-side button options only when the options are short (2 to 4 options max).
Use vertical stacked options when:
- there are 5 or more options
- answers are longer
- questions are more technical
- mobile usability would suffer

So:
- Yes / No / Partial / Don’t know = side by side is fine
- longer descriptive choices = vertical stacked

ADD “DON’T KNOW”
Yes, add “Don’t know” to technical questions where many non-technical users may not know.
This is important.

Examples where “Don’t know” should appear:
- firewall
- backup testing

SCORING NOTE:
